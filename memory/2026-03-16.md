## OpenClaw Gateway 定时任务与 Mac 睡眠

**问题：** Mac 盖盖子睡眠时，定时任务有时触发有时不触发

**分析：**
- Gateway 是 Mac 上的守护进程（launchd 服务）
- 盖盖子睡眠分类型：浅睡眠（CPU继续运行）vs 深度睡眠
- 浅睡眠时 Gateway 仍在运行，任务可触发
- 深度睡眠时任务不触发

## LLM 并发优化（重大改进）

### 问题
Pipeline 批量处理简历时，LLM 评估串行执行，耗时过长。

### 根因
通过 Gateway 的 `openclaw agent` 调用 LLM，但所有请求使用同一个 session（`agent:interviewer:main`），Gateway 对同一 session 的请求串行处理。

### 解决方案
1. 新增 `core/llm_client.py` — 直接调用 MiniMax API
2. 修改 `core/reviewer.py` — 添加两种模式：
   - 默认：Gateway 模式（保持兼容）
   - 直接模式：设置 `RECRUITER_USE_DIRECT_LLM=1` + 配置 API key

### 配置方法
在 `config.local.json` 添加：
```json
{
  "llm": {
    "apiKey": "your-minimax-api-key"
  }
}
```

运行时启用：
```bash
RECRUITER_USE_DIRECT_LLM=1 python test_all_emails.py 10
```

### 效果对比
| 模式 | 10封邮件耗时 | 预估100封 |
|------|-------------|-----------|
| Gateway (串行) | 160秒 | ~27分钟 |
| Direct LLM (5并发) | 37.8秒 | ~6分钟 |
| **提升** | **~76%** | **~76%** |

## 飞书交互 vs Pipeline 批处理

| 场景 | 并发方案 | 说明 |
|------|---------|------|
| 飞书用户交互 | 单session串行 | 需要上下文连续性，结果有序汇总 |
| Pipeline定时任务 | 多session/直接API | 可并发，结果发固定目标 |

飞书用户主动批量处理时，应单session串行+汇总返回，避免结果乱序。

## runtime 目录结构

```

├── cache/parsed/       ← 解析后的PDF缓存（复用解析）
├── incoming/           ← 原始邮件下载
├── logs/               ← 日志
├── outbox/             ← 待发送
├── parsed/             ← 配置定义但未使用（可删除）
├── processed/          ← 评估通过的候选人（通过后写入）
├── reports/            ← 测试报告
│   ├── single-evaluations/{uid}.json  ← 评分结果
│   └── test-run-report.json
└── state/
    └── mail-header-    ← Header索引
```
